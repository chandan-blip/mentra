# Module 05 — Personalized Roadmap

> Phase 2 · First module of Phase 2 · Depends on `01-auth`, `02-user-profile`, `03-assessment` · Blocks `06-daily-tasks`, gives `07-content-delivery` its anchor

---

## 1. Purpose

Turn the student's skill matrix + goals into a **multi-week learning plan**.

- One active roadmap per student
- Composed of weeks, each containing items (topics / projects / assessments / sessions)
- Generated by a **pluggable strategy** — naive default for MVP, user supplies custom algo later
- Regeneratable when inputs change (new assessment, changed goals, changed available hours)
- Item completion can be marked manually now; auto-completion will hook in once `08-progress-tracking` and `07-content-delivery` ship

This module is the **plan**. `06-daily-tasks` is the day-by-day execution layer that pulls from this plan.

## 3. User Flows

### 3.1 First-time generation (auto, after assessment)
1. `03-assessment` emits `assessment.completed` for the `initial` template.
2. `roadmap.listener` consumes the event.
3. Calls the active generator strategy with `{ profile, skillMatrix, assessmentAttempt }`.
4. Strategy returns a roadmap plan (weeks + items).
5. Persists as a new `Roadmap` with `status = active`. Any previous active roadmap for this user flips to `archived`.
6. Emits `roadmap.generated`.
7. Dashboard widget unlocks on next overview load.

### 3.2 Manual regeneration
1. Student clicks "Regenerate roadmap" in the FE.
2. FE → `POST /roadmap/me/regenerate`.
3. BE runs the same flow as 3.1 but with the latest inputs (current skill matrix, current profile).
4. Returns the new roadmap.

### 3.3 Auto-regeneration triggers
The module listens for:
- `assessment.completed` for any periodic/topic template → re-evaluate.
- `student-profile.updated` where `changedFields` ∩ `{ goal, targetRoles, studyHoursPerDay, techStack }` is non-empty.

Throttle: at most one auto-regeneration per user per 6 hours (Redis-locked). Manual regeneration ignores throttle.

### 3.4 Item completion
- Student marks an item complete manually from the FE.
- `POST /roadmap/items/:id/complete` toggles status, sets `completedAt`.
- Emits `roadmap.item.completed` — consumed later by progress-tracking + gamification.
- Optional unlock semantics: if `RoadmapItem.dependsOnIds` is set, completion of dependencies unlocks downstream items (auto-status `available`).

### 3.5 Week view (FE landing for `/roadmap`)
- Default landing is the current week (`current` = earliest `RoadmapWeek` with a not-yet-completed item, or week 1).
- Scrubber lets the student navigate weeks; future weeks are visible but visually marked as upcoming.

## 4. Data Model

```prisma
model Roadmap {
  id              String         @id @default(cuid())
  userId          String
  status          RoadmapStatus  @default(active)
  generatedBy     String         // strategy id used, e.g. "default-v1"
  basisAttemptId  String?        // assessment attempt that drove this generation
  basisProfileVersion Int?       // bump on profile changes that triggered regeneration
  totalWeeks      Int
  startedOn       DateTime       @default(now())
  archivedAt      DateTime?
  notes           String?        @db.Text
  weeks           RoadmapWeek[]

  @@unique([userId, status], name: "one_active_roadmap_per_user")
  @@index([userId])
}

model RoadmapWeek {
  id          String         @id @default(cuid())
  roadmapId   String
  weekNumber  Int
  title       String
  theme       String?
  startsOn    DateTime?      // calendar planning; nullable for "self-paced" mode
  endsOn      DateTime?
  roadmap     Roadmap        @relation(fields: [roadmapId], references: [id], onDelete: Cascade)
  items       RoadmapItem[]

  @@unique([roadmapId, weekNumber])
  @@index([roadmapId])
}

model RoadmapItem {
  id            String             @id @default(cuid())
  weekId        String
  order         Int                // position within the week
  type          RoadmapItemType
  title         String
  description   String?            @db.Text
  skillIds      String[]           // tags into Skill table from assessment module
  estimatedMin  Int?               // rough effort, used by daily-tasks splitter
  contentRef    Json?              // { kind: "lesson"|"module"|"external", id|url }
  dependsOnIds  String[]           // other RoadmapItem ids that must complete first
  status        RoadmapItemStatus  @default(locked)
  completedAt   DateTime?
  week          RoadmapWeek        @relation(fields: [weekId], references: [id], onDelete: Cascade)

  @@unique([weekId, order])
  @@index([weekId, status])
}

enum RoadmapStatus      { active archived superseded }
enum RoadmapItemType    { topic project assessment session reading practice }
enum RoadmapItemStatus  { locked available in_progress completed skipped }
```

### Status transitions (item)
```
locked  ──deps met──►  available  ──user starts──►  in_progress  ──complete──►  completed
                              \                                            /
                               \────────────────skip──────────────►  skipped
```

A new roadmap starts with week-1 items as `available` (no deps), all later items as `locked` until their deps complete. The generator is responsible for declaring `dependsOnIds`.

## 5. API Endpoints

All under `/api/v1/roadmap`, all require `requireAuth` + `requireOnboardingComplete`.

| Method | Path | Body / Query | Returns |
|---|---|---|---|
| GET | `/me` | — | Active roadmap with weeks + items (status, completion %) |
| GET | `/me/week/:n` | — | Single week with items |
| GET | `/me/summary` | — | `{ totalWeeks, completedItems, totalItems, percentComplete, currentWeek }` |
| POST | `/me/regenerate` | `{ reason?: string }` | New active roadmap |
| GET | `/me/history` | `?limit=10` | Past `archived` / `superseded` roadmaps (metadata only) |
| GET | `/me/history/:roadmapId` | — | Full archived roadmap (read-only) |
| POST | `/items/:id/start` | — | `{ status: "in_progress" }` |
| POST | `/items/:id/complete` | — | `{ status: "completed", unlocked: [itemId,...] }` |
| POST | `/items/:id/skip` | `{ reason?: string }` | `{ status: "skipped" }` |

### Validation
- Item transitions enforced server-side. e.g. cannot complete a `locked` item.
- All item writes scoped to the authenticated user's active roadmap.
- Cannot mutate items in `archived` / `superseded` roadmaps.

### Pluggable generator
- `generator.interface.ts` is the seam:
  ```ts
  interface RoadmapGenerator {
    id: string;
    generate(input: GeneratorInput): Promise<RoadmapPlan>;
  }
  ```
- `GeneratorInput`: `{ userId, profile, skillMatrix, lastAttempt, availableHoursPerWeek, goal, targetRoles, techStack }`.
- `RoadmapPlan`: a serializable description of weeks + items with `dependsOnIds` referenced by *local* item keys (resolved to DB ids on persist).
- Strategy chosen via env var `ROADMAP_GENERATOR_STRATEGY=default | weighted | custom`. Registry pattern; one-line add to register a new generator.

### Default generator (MVP) — high level
- Compute target weeks from `studyHoursPerDay` × 7 vs total effort budget (cap 8–12 weeks for the initial plan).
- Bucket skills by weakness (score < 40 = high priority, 40–70 = medium, > 70 = polish).
- Lay out high-priority skills first; introduce one project around week 4 and another around week 8.
- Each week gets 4–6 items totaling roughly the student's weekly hours.
- Cross-skill dependencies: e.g. "React Hooks" depends on "JavaScript fundamentals" if both are in the plan.

### Events
- **Listens:** `assessment.completed`, `student-profile.updated`.
- **Emits:** `roadmap.generated`, `roadmap.archived`, `roadmap.item.started`, `roadmap.item.completed`, `roadmap.item.skipped`.

### Cross-module exports
- `getActiveRoadmap(userId)` — used by `06-daily-tasks` to source today's items.
- `getRoadmapSummary(userId)` — used by `04-dashboard` widget.

## 7. Security

- All endpoints scoped to `me`. Mentors/admins read other students via internal repository (not REST surface) — that's the mentor module's job.
- Generator runs as a background-safe operation; bounded execution (5s soft timeout) so a bad strategy can't hang requests.
- Manual regeneration rate-limited: 5/hour/user. Auto-regenerations Redis-locked per user, 6-hour TTL.
- Item-completion writes audited (Pino structured logs).

## 10. Testing

- **Unit**:
  - Default generator produces N weeks within the user's hour budget.
  - Item status machine rejects invalid transitions.
  - Dep resolver unlocks correct items on completion.
- **Integration**:
  - `assessment.completed` event creates a roadmap end-to-end.
  - Manual regeneration archives the previous active roadmap atomically.
  - Profile update with non-impactful field does NOT regenerate; with impactful field does.
  - Throttle blocks too-frequent auto-regenerations.
  - Cannot complete a `locked` item.
  - Cannot write to an archived roadmap.
- **Property**: generator output round-trips through persist + read with identical structure.

Coverage target: 85%+ on service + generator + transitions.

## 14. Runbook

- **"Roadmap empty after assessment"** → check `assessment.completed` listener subscribed; check Pino logs for `roadmap.generation.requested` near completion time; manually re-trigger via `POST /roadmap/me/regenerate`.
- **"Generation slow / timing out"** → custom strategy regressed; flip `ROADMAP_GENERATOR_STRATEGY=default` to fall back. Inspect failed metric for offending strategy id.
- **"Student stuck on locked item"** → check `dependsOnIds`; if a generator emitted a bad cycle, re-run regeneration.
- **"Wrong active roadmap shown"** → unique-active constraint guarantees one row; if more appear, the constraint migration didn't run — audit `prisma migrate status` in prod.
- **Rollback**: archive operation is reversible; flip the active status back manually if a regeneration was a mistake. No destructive ops.

## 16. Open Questions

1. **Calendar dates vs self-paced** — should weeks have real calendar dates (`startsOn`, `endsOn`) from day 1, or stay self-paced? Calendar dates enable deadline pressure + reminders, but require pacing decisions per student. Flag exists; default off for MVP.
2. **Plan length** — cap at 8 or 12 weeks for the initial roadmap? Longer plans risk obsolescence; shorter plans need frequent regeneration. Suggest 12.
3. **Skip vs. complete** — should skipping count toward completion %, or be ignored? Currently ignored. If counted, encourages skipping; if not, frustrates students who legitimately don't need an item.
4. **Roadmap diff on regeneration** — when regenerating, do we (a) replace wholesale and archive, (b) preserve completed items as "carried over", or (c) compute a diff and only update changed weeks? MVP picks (a) for simplicity.
5. **Item-content linkage** — `contentRef` is JSON-typed for forward compat. Should it become a typed FK to `Lesson` once `07-content-delivery` ships? Recommendation: yes, via a future migration that promotes `contentRef.lessonId` → `lessonId` column.
